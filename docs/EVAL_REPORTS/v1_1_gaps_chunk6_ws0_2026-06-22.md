## 2. Why a rich cache (the chunk-6 cache boundary)

The chunk-5 probe cached only the final `FrameFingering`. Its `(4, 6, 25)` logit
grid is *not* argmaxed, so the four discrete orientation flips were re-runnable
cache-only — but a discrete flip cannot undo GAPS's *graded* −1..−4 string offset,
which is why the chunk-5 cache-only sweep found no win. Everything that *could*
fix the string axis (per-clip board calibration, image-cued orientation,
perspective-robust projection) lives **upstream** of that grid, in
`predictions_to_homography` and `compute_fingering`, whose inputs the chunk-5
cache discarded. So each geometry iteration cost a full CV re-run.

The v2 cache persists those inputs — `OBBPredictions` (nut/fret/neck anchors, the
WS1 calibration cues), the fitted `Homography`, and the selected `HandSample` —
and reconstructs the fingering via `fingering_from_raw`
(`replace(compute_fingering(hand, H, cfg), t=t)`). Geometry/orientation/posterior
changes now re-derive from cache. **Bonus:** the rich cache is *smaller* than the
legacy one (6.2 MB vs 35 MB over the clean-12; 027: 1.57 MB vs 6.98 MB) because it
stores compact raw inputs instead of a dense float64 grid per frame.

## 4. Reproduce

```bash
cd tabvision
export TABVISION_DATA_ROOT=~/.tabvision/data
export PATH=~/.tabvision/tools/ffmpeg-master-latest-win64-gpl/bin:$PATH
# one-time: build the v2 rich cache (CV re-run at 360p; audio already cached)
python -m scripts.eval.v1_1_gaps_video_chain_probe \
  --checkpoint ~/.tabvision/data/models/guitar-yolo-obb-finetuned.pt --vote-frames 1
# leading indicator (cache-only, seconds):
python -m scripts.eval.v1_1_gaps_string_diag
```

Rich cache: `{stem}.rawcv.c0.25.pkl` under `~/.tabvision/cache/gaps_video_chain/`
(the legacy `{stem}.frames.c0.25.pkl` remains a read-only fallback).
