## 2. Diagnosis (confirmed from the cache)

Reproduced from `~/.tabvision/cache/gaps_video_chain/` (the `_diag.log` cache-only analysis:
predicted vs gold string per ambiguous note, best of the 4 fixed orientations per clip, gold pitch):

- **Ambiguous-note string accuracy = 4178/7697 = 0.543** (chance ~0.33–0.50 for k=2–3 candidate
  strings). The signal is barely above chance.
- **pred−gold string-offset histogram** (best-orient, all clips):
  `−5:7 −4:168 −3:585 −2:539 −1:1716 +0:4179 +1:450 +2:53`. Errors are **systematically biased
  negative** (predicted string-index < gold), spread over **−1..−4** — *not* a single clean flip
  and *not* a single constant offset. (`string_idx` 0 = low-E/bass; negative = predicting toward
  the bass side.)
- **pred−gold fret-offset histogram:** `−5:303 −4:147 +0:4179 +4:523 +5:1193`. Fret errors
  **mirror** the string errors at ±4/±5 (a pitch-preserving move: one string toward the bass ≈ 5
  semitones, 4 for the G–B pair, compensated on fret). **So fixing strings fixes frets** — this is
  one axis error, not two.
- **Per-clip best fixed orientation varies** (`none` / `flip-string` / `flip-both`), and the
  cache-only sweep (chunk-5 §4) shows **under no gate/orientation setting does video beat
  audio-only**; ungated with the per-clip *oracle* orientation it still hurts 10/12 (mean −0.052).

**Decomposition into two coupled failure layers:**

1. **A systematic axis/anchoring error** (the dominant component). The monotone-negative string
   bias + the pitch-preserving fret mirror is what a per-clip **cross-string axis that is
   mis-anchored/mis-oriented** produces. The discrete 4-flip orientation group *cannot* undo a
   graded −1..−4 offset (a flip is a reflection `s→5−s`, not a shift). This is geometry, and it is
   where most of the 0.46 string-error mass sits.
2. **A thin per-string margin** (the residual jitter component). At 360p the inter-string spacing
   is only ~15–22 px (estimate), while the posterior's string kernel is tight
   (`sigma_string_cells = 0.6` → σ_y ≈ 0.10 of board width ≈ ~10 px at 360p). YOLO-corner +
   MediaPipe-tip jitter is on the order of one σ, so adjacent-string confusion is easy even when
   the axis is right.

Root cause (chunk-5 §6, now grounded in code): the fingertip→string geometry is **imprecise and
perspective-biased**, and the board orientation/calibration was **tuned to the non-mirrored Kaggle
UT-Austin rig**; neither transfers to diverse in-the-wild classical-guitar footage. It is *not* a
clean orientation flip nor a constant offset, so chunk-6 must improve the geometry itself, not
re-tune fusion.

## 3. Where the failure lives in the code (grounded map)

| Layer | File | What it does today | Rig assumption baked in |
|---|---|---|---|
| Neck/nut/fret detection | `video/guitar/yolo_backend.py` (`predict_all`, `OBBPredictions` :91-108) | Emits `neck` (center, w/h, rotation), `nut` (center **+ unused edge**), and a **sequence of `fret` OBBs (centers/orientations) — currently unused for geometry**, only counted for a +0.05 confidence bonus | detector recall tuned on Roboflow data; fewer nut/fret boxes on GAPS |
| Homography / canonical board | `video/fretboard/keypoint.py` (`predictions_to_homography` :72-106) | 4-point fit: unit square ↔ neck-OBB corners; `_order_corners_by_neck_anatomy` (:137-199) uses **smaller-mean-X = nut/headstock-left**; `_orient_string_axis_for_lap_framing` (:240-254) flips high-E/low-E by **nut-vs-body X**; `_extend_nut_edge_to_detection` (:202-237, `min_extension_px=5.0`, a **resolution-dependent px constant**) | headstock-left, high-E-up, iPhone-lap framing — the Kaggle diagonal |
| Confidence | `keypoint.py:99-104` | `base_conf + nut_bonus(0.05) + fret_bonus(0.05)` — **detection-quality only** | a confidently-detected but *mis-oriented* board scores high → passes the gate while wrong |
| Fingertip → (string,fret) | `video/hand/fingertip_to_fret.py` (`compute_fingering` :95-172) | Projects the **raw fingertip landmark** (not the pressing point) through `H_inv`; Gaussian over **uniform** fret cells (`fret_xs` :126, not rule-of-18) and string centers (`string_ys` :129); **no finger-width, no pressing-point offset, no temporal/contact geometry**; `PosteriorConfig` σ/curl/z constants (:63-87) tuned to the rig | tip≈contact, uniform frets, rig-scale σ |
| Orientation / vote / gate | `fusion/vision_evidence.py` | `choose_orientation` (:134-155) picks 1 of 4 flips by **audio-pitch-symmetric** support (can't break the pitch-preserving mirror); `combine_fingerings` (:77-107) flat-averages marginals; `gate_fingering_to_audio` (:158-194) + the 0.71 clip-coverage kill-switch (probe `_assemble_fingerings` :303-305) | the runtime band-aid for the bakes-in above |

**The cache boundary (decisive for iteration cost).** The probe caches the **final**
`FrameFingering` per frame in `{stem}.frames.c{conf}.pkl` (`_raw_frame_cache` :229-258). That grid
(`finger_pos_logits`, shape `(4,6,25)`, **not** argmaxed) survives to fusion, so:

- **Cache-only re-runnable** (seconds, no CV): everything in `vision_evidence.py` — orientation
  flips, voting, all three gate thresholds, the 0.71 coverage floor. But these can only apply one
  of **4 discrete whole-grid flips**, which cannot undo the graded −1..−4 offset. (This is exactly
  why chunk-5's cache-only sweep found no win.)
- **NOT cached, so NOT cache-only today:** the canonical fingertip points `(cx,cy)`, the raw
  `HandSample`, the `Homography`, and the `OBBPredictions` anchors (nut/fret/neck) are all computed
  and then **discarded before pickling**. Therefore *any* fix to the projection or the per-clip
  board geometry/orientation currently costs a **full CV re-run** (MediaPipe + YOLO, ~0.5 s/frame ×
  hundreds–thousands of frames × 12 clips ≈ tens of minutes each iteration).

This boundary is the single most important fact for chunk-6 sequencing, and it motivates WS0.

## 4. The cheap-iteration enabler (why WS0 comes first)

Every workstream that matters (per-clip calibration, orientation, projection) touches geometry
*upstream* of the cached grid. Without intervention, each iteration is a full CV re-run. The fix is
a **one-time** investment: extend the probe's frame cache to persist the **raw CV intermediates**
per needed frame — `(OBBPredictions anchors, HandSample, Homography)` — and refactor the probe so
the homography-fit → orientation → `compute_fingering` → fusion stages all run **from the cache**.

After one CV re-run to build this richer cache (v2), every subsequent geometry/orientation/posterior
experiment is **seconds-to-minutes**, exactly as fusion tuning is today. This is a probe/eval-harness
change only — it does **not** touch any `tabvision/` package code or any §8 contract (the anchors are
already public on `YoloOBBBackend.predict_all` and `compute_fingering` already takes `HandSample` +
`Homography`). It ships with tests and reproduces the chunk-5 numbers exactly as its acceptance gate.

## 5. Prioritized workstreams

Ordered by **diagnostic value per unit cost**, attacking the dominant *systematic* error first.
This reorders the prompt's listing (which led with 720p): 720p is expensive (re-download + full CV
re-run) and, per the diagnosis, addresses only the *smaller* jitter component — so it is deferred to
WS3, behind the geometric fixes that attack the dominant systematic bias. (This reordering is the
first item to confirm in §10.)

Each workstream lists: goal · what changes · cache-only after WS0? · gate · cost/risk.

### WS0 — Rich CV-intermediate cache + reusable diagnostic *(enabler)*
- **Goal:** make WS1/WS2/WS4 cache-only; codify the `_diag.log` analysis into a reusable script.
- **Changes:** `scripts/eval/v1_1_gaps_video_chain_probe.py` — capture `yolo.predict_all(frame)`
  anchors + the `HandSample` + `Homography` alongside each frame; new versioned cache file
  (`{stem}.raw.v2.pkl`); split `_raw_fingering_for_frame` so fit/orient/project run from the cache.
  New `scripts/eval/v1_1_gaps_string_diag.py` (cache-only) emitting ambiguous-string accuracy +
  pred−gold string/fret offset histograms (the `_diag.log` table).
- **Cache-only after WS0:** n/a (this builds it). One CV re-run at current 360p.
- **Gate:** from the v2 cache, the probe reproduces chunk-5 exactly (gold 0.8148 / 0.8148 / 0.9726;
  highres 0.7612 / 0.7612 / 0.9099) and the diagnostic reproduces **0.543**. Unit tests for the new
  cache round-trip + the diagnostic math.
- **Cost/risk:** one ~tens-of-minutes CV re-run; no new deps; low risk (harness-only).

### WS1 — Per-clip geometric fretboard calibration *(training-free; attacks the systematic bias)*
- **Goal:** anchor the canonical board's string/fret axes to **detected per-clip cues** instead of
  the fixed Kaggle rig — directly targeting the monotone-negative string bias + ±4/±5 fret mirror.
- **Changes (impl-side, behind `FretboardBackend`):** in `keypoint.py`, use the **ordered fret-OBB
  sequence** to set true (nonlinear) fret x-positions and to disambiguate nut-vs-body by
  spacing-decay (frets compress toward the body); use the **nut-OBB edge** (orientation + extent,
  not just its center) to set the cross-string axis; make `confidence` **geometry-aware**
  (reprojection consistency of the fret OBBs against the fitted board) so per-clip selection/gating
  is trustworthy. Optionally feed the nonlinear fret map into `compute_fingering`'s `fret_xs`.
- **Cache-only after WS0:** yes (anchors are in the v2 cache; re-fit + re-project run from it).
- **Gate:** ambiguous-string accuracy ↑ (toward 0.75); gold `+real(auto)` ↑ with **no per-clip
  regression**; pred−gold histograms recentre on 0.
- **Cost/risk:** medium effort; **overfitting risk** — calibration must use *only* per-clip detected
  cues (no per-clip-tuned constants), since clean-12 is the only video+gold corpus (n=12). The
  bootstrap lower-95 is the honest gate; watch per-clip variance, not just the mean.

### WS2 — Continuous / image-cued auto-orientation *(replaces the discrete 4-flip guess)*
- **Goal:** resolve nut-vs-body and high-E/low-E from the **image**, per clip, replacing both the
  Kaggle-baked heuristics in `keypoint.py` and the pitch-symmetric `choose_orientation`.
- **Changes:** derive nut-vs-body from fret-spacing decay (WS1 already computes it); derive
  high-E/low-E from a per-clip image cue (e.g. fretting-hand position relative to the neck axis, or
  string-thickness gradient) rather than `nut_mid.x` vs `body_mid.x`; have the probe choose
  orientation from these cues with confidence, falling back to audio-support only on ties.
- **Cache-only after WS0:** yes.
- **Gate:** auto-chosen orientation agrees with the per-clip best-fixed orientation on ≥ ~10/12;
  `+real(auto)` approaches `+real(best-orient)`.
- **Cost/risk:** medium; tightly coupled to WS1 (both are per-clip board geometry from OBB cues) —
  **may merge** into one "per-clip board calibration" change.

### WS3 — 720p resolution A/B *(expensive; cost decision point — see §8)*
- **Goal:** test whether the residual *jitter* component is resolution-limited, once WS1/WS2 have
  fixed the systematic bias.
- **Changes:** `scripts/acquire/gaps_video.py` — drop the literal `"18/"` (360p progressive) from the
  format string (`:185`) and raise the `bv*+ba/b` fallbacks to `<=720` (forces a DASH-merge → needs
  ffmpeg, already required); **purge & resolution-key the caches** (`{stem}.mp4`,
  `{stem}.raw.v2.pkl`, `{stem}.offset.pkl`) — current filenames are **not** resolution-keyed, so a
  naive re-download is a silent no-op (footgun). Re-derive offsets for hygiene (alignment is on the
  audio track, so the true offset is unchanged).
- **Cache-only:** no — full re-download + CV re-run at 720p.
- **Gate:** clean A/B vs 360p on string accuracy + gold `+real`. Run **only if** WS1/WS2 plateau with
  a margin near σ (i.e. jitter-limited).
- **Cost/risk:** tens of minutes–hours of CV; ~0.5–1 GB download; no new deps. **Decision point.**

### WS4 — Perspective-robust fingertip→string *(the core 0.54 lever; geometry first, training = HARD STOP)*
- **Goal:** improve the projection itself: model the **pressing point** (behind the visible tip,
  toward the nut along the finger axis), finger-width-aware string assignment under perspective, and
  possibly multi-frame contact geometry; re-evaluate the rig-tuned σ/curl/z constants.
- **Changes:** `fingertip_to_fret.py` (`compute_fingering`, `PosteriorConfig`) — all free-to-change
  impl behind `HandBackend`. Re-runs `compute_fingering` from the cached `HandSample` + `Homography`
  (cache-only re geometry thanks to WS0 — **no MediaPipe re-run**).
- **Cache-only after WS0:** yes for the geometry (projection from cached HandSample); a different σ
  also only needs the cached canonical inputs.
- **Gate:** ambiguous-string accuracy → 0.75+; the dominant lever if WS1/WS2 leave residual.
- **Cost/risk:** highest effort. **HARD STOP:** if a *geometric* pressing-point/finger-width model
  caps out below target, the next option is a small **pretrained-backbone fingertip-on-fretboard
  classifier** — that is **training** (data + compute + cost). Per SPEC §0 rules 6 & 8, **stop and
  ask the user before any training run or new dependency.** Do not start it within chunk-6 without
  explicit approval.

### WS5 — Gate / confidence recalibration *(last; cache-only; preserve no-regression)*
- **Goal:** once the evidence is genuinely better, re-derive the per-event gate
  (`min_homography_confidence` / `min_candidate_support` / `min_best_ratio`) and the
  `min_clip_coverage` kill-switch so the lift **materializes** instead of being gated out — without
  re-introducing regression.
- **Changes:** `GateParams` defaults in the probe (and, if promoted to production, the
  `vision_evidence` gate) — re-run the chunk-5 §4 style sweep to re-derive the coverage threshold.
- **Cache-only after WS0:** yes.
- **Gate:** the no-regression invariant (§1) holds at the new thresholds; `+real(auto)` realises the
  WS1/WS4 string-accuracy gains.
- **Cost/risk:** low; the discipline is conservatism (the chunk-5 lesson: 0.5 leaked and regressed
  ~0.05).

## 6. Per-step eval gate (reusing the cached probe)

Two instruments, both run from the cache after WS0 (seconds–minutes):

1. **Headline (lagging) — the canonical 4-condition probe.** `v1_1_gaps_video_chain_probe.py`,
   gold-audio frame: `audio-only / +real(auto) / +real(best-orient) / +oracle`, mean + bootstrap
   lower-95, plus the highres frame as the honest end-to-end. Pass = §1's primary criterion (gold
   `+real(auto)` ≥ 0.865, no per-clip regression) and the no-regression invariant on both frames.
   The probe already computes all of this; chunk-6 changes only what feeds it.
2. **Leading — the cache-only string diagnostic (WS0).** Ambiguous-note string accuracy + pred−gold
   string/fret offset histograms. This is the *fast* signal each geometry change is judged on
   (target ≥ 0.75; histograms recentred on 0) before paying the full fusion/scoring pass.

Every change must report **both** and must not regress audio-only on any clip. Per-clip variance and
the bootstrap lower-95 (not just the mean) are the honest readout given n=12.

## 8. Risks & cost decision points

- **[COST/STOP] Training (WS4 fallback).** A learned fingertip-on-fretboard model is the one path
  that adds data + compute + cost. **Hard stop — ask first** (SPEC §0 rules 6 & 8). Chunk-6 stays
  geometric unless the user approves training as a follow-on.
- **[COST] 720p re-run (WS3).** Re-download + full CV re-run; ~0.5–1 GB; tens of min–hours. No new
  dependency. Decision point: run it only if WS1/WS2 leave a jitter-limited residual, and confirm
  before paying it.
- **[CACHE FOOTGUN] Resolution is not cache-keyed.** `download_video` short-circuits on file
  existence and `{stem}.mp4` / `{stem}.raw.v2.pkl` / `{stem}.offset.pkl` omit resolution. WS3 must
  purge or version these or it silently reuses 360p artifacts.
- **[OVERFIT / small-n] clean-12 is the only video+gold corpus (n=12).** Per-clip calibration must
  use only *detected* cues (no per-clip constants); judge on bootstrap lower-95 and per-clip
  no-regression, not the mean alone.
- **[NO-REGRESSION] invariant.** Video must remain additive: any clip with weak/wrong evidence must
  fall back to audio-only exactly. WS5 re-tunes the gate conservatively; the chunk-5 lesson (0.5
  leaked, −0.05) is the cautionary baseline.
- **[NC / offline-only] GAPS media.** Source mp4s and all caches stay under `~/.tabvision/cache/`;
  never committed or redistributed. (Unchanged from chunk-5.)
- **[DETECTION-LIMITED] geometry-aware confidence depends on YOLO firing nut/fret boxes.** If the
  detector under-fires on some GAPS clips, WS1's reprojection-consistency confidence degrades to the
  neck-only fit; those clips should gate to audio-only, not guess.

## 9. Sequencing & process

1. **WS0** first (enables cheap iteration; gate = reproduces chunk-5 from the v2 cache).
2. **WS1 + WS2** (likely merged: per-clip board calibration + image-cued orientation) — the main
   bet against the systematic bias; iterate cache-only on the leading diagnostic, confirm on the
   headline probe.

   materialise through the gate, then chunk-6 meets its must-pass.
4. If still short and jitter-limited → **WS3** (720p, decision point).
5. If still short and projection-limited → **WS4 geometric** part; the learned-model fallback is a
   **STOP-and-ask**.

Throughout: one workstream at a time (SPEC §0 rule 2); ship tests with every code change; keep
